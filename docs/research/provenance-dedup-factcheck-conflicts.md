## Executive Summary

Noodly's current architecture implements a custom truth pipeline: claims are extracted from source artifacts via LLM (`src/noodly/extraction/extractor.py`), scored with a multiplicative truth score (`src/noodly/models/claims.py:truth_score`), stored in a bitemporal ledger (`src/noodly/scoring/ledger.py`), and projected to Markdown. The system delegates deduplication to Graphiti's built-in hybrid search (`src/noodly/storage/graphiti_backend.py:search_similar_claims`).

This research identifies **five high-impact improvements**, ordered by expected ROI:

1. **Adopt W3C PROV-DM vocabulary** for provenance metadata on `ClaimEvidence` — low effort, high interoperability gains, and makes provenance chains queryable.
2. **Replace the linear-scan deduplication with a FAISS HNSW or pgvector index** — the current Graphiti-delegated search works but is tightly coupled; a dedicated ANN index gives control over thresholds and performance.
3. **Implement a configurable trust-propagation algorithm** (TruthFinder or Investment-style) to replace the fixed multiplicative `truth_score` formula — enables iterative source-reliability estimation.
4. **Add predicate canonicalization** via embedding similarity or a lightweight ontology — reduces semantic duplicates from varied predicate phrasings.
5. **Introduce a formal conflict-resolution framework** inspired by Dung's argumentation semantics and MDM survivorship rules — makes conflict handling explicit, auditable, and configurable per the user's preference for pluggable strategies.

## 1. Provenance

### 1.1 W3C PROV-DM / PROV-O

**Spec:** [https://www.w3.org/TR/prov-dm/](https://www.w3.org/TR/prov-dm/) (W3C Recommendation, April 2013)

#### Data Model

PROV-DM defines three core types and their relationships:

| PROV Type | Description | Noodly Equivalent |
|-----------|-------------|-------------------|
| **Entity** | A physical, digital, or conceptual thing | `Claim`, `SourceArtifact` |
| **Activity** | Something that occurs over a period of time | LLM extraction, ingestion pipeline run |
| **Agent** | Something that bears responsibility | Document author, the extractor model, a human reviewer |

Key relationships:

- **`wasGeneratedBy`** — an Entity was generated by an Activity (a `Claim` was generated by an extraction run).
- **`used`** — an Activity used an Entity (extraction used a `SourceArtifact`).
- **`wasAttributedTo`** — an Entity is attributed to an Agent (a `Claim` is attributed to the document author).
- **`wasDerivedFrom`** — an Entity was derived from another Entity (a canonicalized claim was derived from a raw extraction).
- **`actedOnBehalfOf`** — delegation between Agents.
- **`wasInformedBy`** — Activity-to-Activity communication.

Extended structures include **Bundles** (provenance of provenance) and **Alternate/Specialization** links for entity identity.

#### Mapping to Noodly

Noodly's `ClaimEvidence` (`src/noodly/models/claims.py:33-42`) currently tracks:
- `artifact_id` → maps to PROV `used` (the Activity used this Entity)
- `author` / `author_role` → maps to PROV `Agent` + `wasAttributedTo`
- `source_span` → no direct PROV equivalent; this is a fine-grained locator within an Entity
- `supports` → no PROV equivalent; this is domain-specific polarity

**What's missing in Noodly:**
- No `Activity` record — there is no first-class representation of "the extraction run that produced these claims." Without this, you cannot trace *when* and *how* a claim was extracted, only *from what*.
- No `wasDerivedFrom` chain — when a claim is superseded or merged, the derivation link is lost. The `ClaimStatus.superseded` flag exists but doesn't point to the successor.
- No delegation tracking — if a human reviewer promotes a claim (`owner_confirmed`), the reviewer Agent is not recorded.
- No Bundle support — provenance of the provenance itself (e.g., "which version of the extractor prompt produced this claim?") is not tracked.

#### Strengths
- W3C standard with broad tooling support (PROV-N, PROV-JSON, PROV-O for RDF).
- Interoperable with other systems that speak PROV.
- Well-defined formal semantics with inference rules.

#### Weaknesses
- Can be verbose for simple use cases.
- The generic vocabulary doesn't cover domain-specific concepts (polarity, confidence scores).
- RDF-based tooling (PROV-O) may be overkill for a system not using RDF.

#### Recommendations for Noodly
- **Add an `ExtractionRun` model** (PROV Activity) linking to the extractor version, prompt hash, and timestamp. Reference it from `ClaimEvidence.episode_id`.
- **Add `derived_from: UUID | None` to `Claim`** to track supersession chains explicitly (PROV `wasDerivedFrom`).
- **Record the reviewer Agent** when a claim is promoted via `promote_claim()` in `src/noodly/scoring/ledger.py:152`.
- Use PROV-JSON as a serialization format for provenance export, enabling downstream auditing.

### 1.2 Content-Addressable Storage / Merkle DAGs

#### Core Concept

Content-addressable storage (CAS) identifies data by its cryptographic hash rather than by location. A **Merkle DAG** (Directed Acyclic Graph) extends this: each node's identifier is a hash of its content plus the hashes of its children, creating a tamper-evident tree.

**Key systems:**
- **Git** — commits, trees, and blobs are all content-addressed. A commit hash covers the entire repository state at that point.
- **IPFS / IPLD** — generalizes Merkle DAGs across protocols. Content Identifiers (CIDs) are self-describing hashes. Properties: immutability, deduplication, authentication, and disconnected-first operation.
- **Sigstore** — uses content hashing and transparency logs for software artifact provenance. Artifacts are signed and the signatures are recorded in an append-only log (Rekor).

#### Algorithm / Data Model

```
CID = hash(content + links_to_child_CIDs)
```

Properties:
- **Immutability**: changing any byte changes the CID.
- **Deduplication**: identical content has the same CID, stored once.
- **Authentication**: a client can verify data by recomputing the hash.
- **Composability**: DAGs can reference sub-DAGs across systems.

#### Applicability to Noodly

| Aspect | Applicability | Notes |
|--------|--------------|-------|
| Artifact integrity | **High** | Hash `SourceArtifact.body` to detect tampering or track exact versions. |
| Claim provenance chain | **Medium** | A Merkle chain of claim versions would make the supersession history tamper-evident. |
| Deduplication | **Low** | Content-hash dedup is exact; Noodly needs *semantic* dedup. |
| Storage backend | **Low** | IPFS as a storage backend adds complexity without clear benefit for an internal knowledge system. |

#### Strengths
- Tamper-evident provenance with zero trust required.
- Natural deduplication at the storage layer.
- Well-understood, battle-tested (Git has been in use since 2005).

#### Weaknesses
- Only handles exact-match deduplication; semantic similarity requires a separate layer.
- Adds hashing overhead (negligible for small artifacts, relevant for large corpora).
- IPFS-specific tooling is complex and may be overkill for an internal system.

#### Recommendations for Noodly
- **Add a `content_hash: str` field to `SourceArtifact`** (`src/noodly/models/artifacts.py`) computed as `sha256(body)`. Use it as a fast exact-duplicate check before running LLM extraction.
- **Do not adopt IPFS** as a storage backend — the complexity is not justified for Noodly's use case. The content-hash field achieves the key benefit (integrity verification) without the infrastructure overhead.

### 1.3 SLSA Framework

**Spec:** [https://slsa.dev/](https://slsa.dev/) (v1.2 current)

#### Core Concept

SLSA (Supply-chain Levels for Software Artifacts) is a security framework defining increasing levels of provenance trustworthiness for software artifacts. It specifies *what* provenance metadata must be recorded and *how* tamper-resistant the recording process must be.

**Levels (Build Track):**

| Level | Requirements | Focus |
|-------|-------------|-------|
| L0 | None | — |
| L1 | Provenance exists showing how the artifact was built | Mistakes, documentation |
| L2 | Signed provenance from a hosted build service | Tampering after build |
| L3 | Hardened, isolated build service | Tampering during build |

**Provenance schema** (in-toto attestation format):
- `buildDefinition` — what was built and how (source URI, build parameters).
- `runDetails` — the builder identity (`builder.id`), build metadata.
- `subject` — the artifact(s) produced, identified by digest.

#### Relevance to Noodly

SLSA's patterns are directly transferable to knowledge artifact provenance:

| SLSA Concept | Noodly Analogue |
|--------------|----------------|
| Software artifact | `Claim` or projected Markdown file |
| Build platform | The extraction pipeline (`ClaimExtractor` + LLM) |
| Source | `SourceArtifact` |
| Provenance attestation | Could be a signed record: "this Claim was extracted from this Artifact by this model version at this time" |

**Key insight:** SLSA's model of "the builder is a trusted entity that produces provenance" maps cleanly to Noodly's pipeline. The extractor is the builder; the LLM model version and prompt hash are the build parameters; the claims are the artifacts.

#### Strengths
- Provides a graduated maturity model (levels) for provenance completeness.
- Well-defined attestation format (in-toto) with tooling.
- Focuses on tamper-resistance, which is relevant for enterprise knowledge systems where audit trails matter.

#### Weaknesses
- Designed for software supply chains, not knowledge systems — some concepts don't map (e.g., reproducible builds).
- The signing infrastructure (Sigstore, Fulcio, Rekor) is heavy for an internal tool.

#### Recommendations for Noodly
- **Adopt the SLSA provenance *vocabulary* without the signing infrastructure.** Record `builder_id` (model name + version), `source_uri`, and `build_parameters` (prompt hash, temperature) alongside each extraction run.
- This is complementary to the PROV-DM recommendation above — SLSA provides the schema for the Activity's metadata.

### 1.4 OpenLineage / Apache Atlas

#### OpenLineage

**Spec:** [https://openlineage.io/](https://openlineage.io/) (v2.0.2, Apache 2.0 license)

OpenLineage is an open standard for data lineage metadata collection. Its core model:

- **Job** — a process that consumes and produces datasets (identified by namespace + name).
- **Run** — an instance of a Job with a start/completion time and a globally unique run ID (UUID).
- **Dataset** — an abstract representation of data (identified by namespace + name).
- **RunEvent** — an observation of a Run's state (START, RUNNING, COMPLETE, FAIL, ABORT).
- **Facets** — extensible metadata attached to Jobs, Runs, and Datasets (schema facets, column lineage facets, etc.).

**Lineage model:** Jobs consume input Datasets and produce output Datasets. A lineage graph is woven together from observations across many Jobs.

#### Apache Atlas

Apache Atlas is a data governance and metadata management framework for Hadoop ecosystems. It provides:
- A type system for metadata (entities, classifications, relationships).
- REST APIs for lineage queries.
- Integration with Hadoop tools (Hive, Spark, Kafka).

#### Mapping to Noodly

| OpenLineage Concept | Noodly Mapping |
|--------------------|----------------|
| Job | Extraction pipeline, projection pipeline |
| Run | A specific pipeline invocation (`Pipeline.run()` in `src/noodly/pipeline.py`) |
| Input Dataset | `SourceArtifact` collection (e.g., a watched directory) |
| Output Dataset | Claims in the ledger, projected Markdown files |
| Facets | Extraction metadata (model, prompt version), quality scores |

#### Strengths
- Standard JSON schema with SDKs in Python, Java, and others.
- Extensible via custom facets — can carry Noodly-specific metadata (truth scores, knowledge class).
- Growing ecosystem of integrations (Airflow, Spark, dbt).

#### Weaknesses
- Designed for data pipeline lineage (ETL jobs), not knowledge graph provenance — the Job/Dataset abstraction is coarser than PROV's Entity/Activity/Agent.
- Apache Atlas is heavyweight (requires Kafka, HBase, Solr) — not suitable for Noodly's lightweight architecture.
- OpenLineage events are oriented toward batch processing, not the fine-grained claim-level provenance Noodly needs.

#### Recommendations for Noodly
- **Do not adopt OpenLineage or Apache Atlas directly** — they are designed for a different scale and use case (data pipeline governance).
- **Borrow the Facet extensibility pattern** from OpenLineage: define a `ProvFacet` base class that can be attached to Claims and Artifacts, allowing plugins to add domain-specific provenance metadata without changing core models.

## 2. Deduplication

### 2.1 MinHash / LSH (Locality Sensitive Hashing)

#### Core Algorithm

**MinHash** estimates the Jaccard similarity between two sets in constant time:

1. Represent each document as a set of shingles (n-grams).
2. Apply `k` independent hash functions to produce a signature of `k` minimum hash values.
3. The fraction of positions where two signatures agree approximates the Jaccard similarity.

**LSH (Locality Sensitive Hashing)** makes MinHash sub-linear for search:

1. Divide the `k`-element signature into `b` bands of `r` rows each (`k = b × r`).
2. Two documents are candidate pairs if they agree in *all* rows of *any* band.
3. The probability of becoming a candidate pair follows an S-curve: `P = 1 - (1 - s^r)^b` where `s` is the true Jaccard similarity.
4. The threshold is approximately `t ≈ (1/b)^(1/r)`.

**Key library:** [datasketch](https://github.com/ekzhu/datasketch) (Python, MIT license, ~2.9K stars, ~250K daily downloads). Provides `MinHash`, `MinHashLSH`, `MinHashLSHForest`, `WeightedMinHash`, and HNSW.

#### Strengths
- Sub-linear query time: O(1) amortized per query with LSH index.
- Well-understood theoretical guarantees on false positive/negative rates.
- `datasketch` is production-ready with Redis backend support for persistence.
- Excellent for detecting near-duplicate *documents* (e.g., detecting re-ingested artifacts).

#### Weaknesses
- Operates on set similarity (Jaccard), not semantic similarity — "the CEO is John" and "John serves as Chief Executive Officer" have low shingle overlap.
- Requires tokenization choices (shingle size) that affect quality.
- Not directly applicable to short structured claims (subject-predicate-object triples).

#### Relevance to Noodly
- **Artifact-level deduplication**: Excellent fit. Before running LLM extraction, check if a `SourceArtifact` is a near-duplicate of a previously ingested artifact using MinHash on the body text. This prevents redundant extraction runs.
- **Claim-level deduplication**: Poor fit. Claims are short structured triples; semantic embedding similarity is a better approach.

#### Recommendations for Noodly
- **Add a MinHash-based pre-filter to the ingestion pipeline** (`src/noodly/pipeline.py:run`). Before calling `self._extractor.extract(artifact)`, compute the MinHash of `artifact.body` and check the LSH index for near-duplicates above a configurable threshold (e.g., `NOODLY_ARTIFACT_DEDUP_THRESHOLD=0.8`).
- Use `datasketch.MinHashLSH` with a Redis backend for persistence across pipeline runs.
- **Effort: Small** — the `datasketch` library handles the complexity; integration is ~50 lines of glue code.

### 2.2 Approximate Nearest Neighbor (ANN) Indices

#### Overview

ANN indices enable sub-linear similarity search over dense vector embeddings. The key candidates for replacing Noodly's current Graphiti-delegated search:

| Library | Algorithm | Language | Key Properties |
|---------|-----------|----------|---------------|
| **FAISS** (Meta) | IVF, PQ, HNSW, flat | C++ / Python | Highest throughput on GPU; extensive index types; battle-tested at Meta scale. |
| **Annoy** (Spotify) | Random projection trees | C++ / Python | Simple, memory-mapped, read-only after build. Good for static datasets. |
| **ScaNN** (Google) | Anisotropic vector quantization | C++ / Python | Excellent recall/speed trade-off; optimized for inner product similarity. |
| **pgvector** | IVFFlat, HNSW | PostgreSQL extension | Lives inside Postgres; ACID transactions; lower throughput than FAISS but integrated with relational data. |
| **hnswlib** | HNSW | C++ / Python | Pure HNSW; very fast; simple API. |

#### Performance Comparison (from ANN-Benchmarks)

On the standard `glove-100-angular` benchmark (k=10):

| Library | Recall@10 = 0.95 QPS | Index Build Time | Memory |
|---------|---------------------|-----------------|--------|
| hnswlib | ~3000 QPS | Fast | Moderate |
| FAISS HNSW | ~2500 QPS | Fast | Moderate |
| ScaNN | ~4000 QPS | Moderate | Low (quantized) |
| Annoy | ~500 QPS | Fast | Low (mmap) |
| pgvector HNSW | ~200 QPS | Slow | High (in Postgres) |

**Key takeaway:** For Noodly's scale (thousands to tens of thousands of claims, not millions), *all* of these are fast enough. The choice should be driven by integration simplicity and operational concerns.

#### Current State in Noodly

`GraphitiBackend.search_similar_claims()` (`src/noodly/storage/graphiti_backend.py:214-233`) delegates to Graphiti's hybrid search (vector + keyword). This works but:
- Tightly couples deduplication to the Graphiti storage backend.
- Does not work with the `JSONBackend` (no vector search available).
- Does not expose configurable similarity thresholds.
- Returns at most `limit` results without a similarity score, making it hard to set a deduplication threshold.

#### Strengths
- Orders of magnitude faster than linear scan for large corpora.
- Well-maintained libraries with Python bindings.
- HNSW provides a good balance of recall, speed, and incremental insertion.

#### Weaknesses
- Requires maintaining a separate index alongside the primary store.
- Index build/update adds operational complexity.
- Embedding quality determines search quality — garbage in, garbage out.

#### Recommendations for Noodly
- **For the near term (small scale): Use FAISS with a flat (brute-force) index.** At <10K claims, brute-force cosine similarity over embeddings is fast enough and avoids index-maintenance complexity. This can be wrapped in a `DedupIndex` abstraction.
- **For medium scale (10K–1M claims): Switch to FAISS HNSW or hnswlib.** HNSW supports incremental insertion and provides sub-linear search.
- **Make the dedup index backend-agnostic** — decouple it from `GraphitiBackend` so it works with both JSON and Graphiti storage. Add a `NOODLY_DEDUP_BACKEND` config option (`faiss`, `graphiti`, `pgvector`).
- **Expose the similarity threshold as `NOODLY_DEDUP_SIMILARITY_THRESHOLD`** (default: 0.85) so operators can tune dedup aggressiveness.
- **Effort: Medium** — requires an embedding pipeline (can reuse Graphiti's embedder via `Brain.get_embedder()` in `src/noodly/graph/brain.py:55-57`) and a new `DedupIndex` class.

### 2.3 Entity Resolution / Record Linkage

#### Fellegi-Sunter Model (1969)

The foundational statistical model for record linkage. Core idea:

Given two records, compute a **match weight** for each field comparison:

```
weight = log2(m / u)
```

Where:
- `m` = P(field values agree | records match) — e.g., 0.98 for date of birth.
- `u` = P(field values agree | records don't match) — e.g., 1/365 for date of birth.

Sum weights across fields. Classify pairs as:
- **Match** if total weight > upper threshold.
- **Non-match** if total weight < lower threshold.
- **Possible match** (manual review) if between thresholds.

**Key property:** Fields that are rare (low `u`) provide more discriminative power. Matching on "John Smith" is less informative than matching on "Zygmunt Baranowski."

#### Implementations

- **[Splink](https://github.com/moj-analytical-services/splink)** (Python, Spark/DuckDB) — modern, scalable implementation with visualization. Unsupervised EM parameter estimation.
- **[dedupe](https://github.com/dedupeio/dedupe)** (Python, MIT, ~4.5K stars) — active learning approach. Human labels a small sample; the model generalizes. Uses blocking for scalability.
- **[Zingg](https://github.com/zinggAI/zingg)** (Java/Spark) — similar active learning approach, designed for big data.

#### Relevance to Noodly

Entity resolution is relevant for **entity node deduplication** in the knowledge graph — detecting that "Barack Obama", "B. Obama", and "President Obama" refer to the same entity. This is partially handled by Graphiti's built-in entity resolution, but a Fellegi-Sunter approach could improve accuracy.

For **claim deduplication**, the model is less directly applicable because claims are structured triples, not flat records. However, the *field-level comparison* approach is useful: compare subject similarity, predicate similarity, and object similarity independently, then combine weights.

#### Strengths
- Principled probabilistic framework with well-understood properties.
- Active learning (dedupe) reduces manual labeling effort.
- Handles missing data and field-level variation gracefully.

#### Weaknesses
- Designed for structured records with known fields, not free-text or semantic similarity.
- Requires careful blocking strategy for scalability.
- Parameter estimation (m and u probabilities) can be unreliable with small datasets.

#### Recommendations for Noodly
- **Adopt the field-level comparison pattern for claim deduplication.** Instead of comparing claims as flat text, compare `(subject_similarity, predicate_similarity, object_similarity)` independently and combine with learned or configured weights. This is more precise than whole-claim embedding similarity.
- **Do not adopt dedupe/Splink directly** — they solve a different problem (record linkage across datasets). But the conceptual framework is valuable.
- **Effort: Medium** — requires implementing a `ClaimMatcher` that computes per-field similarity and combines them.

### 2.4 Predicate Canonicalization / Ontology Alignment

#### Problem

In open information extraction, the same relationship is expressed many ways:
- "owns", "is the owner of", "is owned by" (inverse), "holds ownership of"
- "works at", "is employed by", "is an employee of", "joined"

Noodly extracts predicates via LLM, which produces varied phrasings. Without canonicalization, semantically identical claims appear as distinct entries.

#### Approaches

**1. WordNet-based synonymy**
- WordNet organizes English words into **synsets** (sets of synonyms).
- Map predicates to WordNet verb synsets; treat two predicates as equivalent if they share a synset or are within a configurable path distance in the hypernym hierarchy.
- **Limitation:** WordNet's verb coverage is limited, especially for domain-specific terms.

**2. ConceptNet**
- A multilingual knowledge graph with commonsense relationships.
- Provides `RelatedTo`, `Synonym`, `IsA` relationships between concepts.
- Can be queried via API or embedded locally.
- **Limitation:** Noisy; relationships are crowdsourced and not always accurate.

**3. Schema.org alignment**
- Schema.org defines a vocabulary of ~800 types and ~1,400 properties.
- For predicates that map to schema.org properties, canonicalize to the schema.org name.
- **Limitation:** Coverage is limited to web-oriented domains.

**4. Embedding-based clustering**
- Embed predicates using a sentence transformer (e.g., `all-MiniLM-L6-v2`).
- Cluster similar embeddings (agglomerative clustering, DBSCAN, or simple threshold-based grouping).
- Select a canonical representative for each cluster (e.g., the most frequent predicate).
- **Strength:** Language-agnostic, handles arbitrary domain vocabulary.

**5. LLM-based normalization**
- During extraction, instruct the LLM to map predicates to a controlled vocabulary.
- Provide the vocabulary in the system prompt or use few-shot examples.
- **Strength:** Leverages the LLM's semantic understanding; can be integrated into the existing extraction step.
- **Weakness:** Adds latency; the controlled vocabulary must be maintained.

**6. Open Knowledge Base Canonicalization (Liu et al., 2022)**
- Joint canonicalization and linking framework (JOCL) using factor graph models.
- Simultaneously canonicalizes noun phrases and relation phrases while linking to a curated KB.
- **Relevance:** Directly addresses the OKB canonicalization problem, but the factor graph approach is complex to implement.

#### Relevance to Noodly

The extraction prompt in `src/noodly/extraction/extractor.py:22-42` does not constrain predicates to a controlled vocabulary. The LLM is free to generate any predicate string, leading to semantic duplicates.

#### Recommendations for Noodly
- **Short term: Add LLM-based predicate normalization to the extraction prompt.** Provide a configurable list of canonical predicates in the system prompt and instruct the LLM to use them when possible. Allow new predicates for relationships not covered by the list. Store the canonical predicate alongside the original for transparency.
- **Medium term: Add embedding-based predicate clustering as a post-processing step.** After extraction, embed all predicates and cluster them. Map each cluster to a canonical form. This catches cases the LLM-based approach misses.
- **Make the canonical predicate vocabulary configurable** via `NOODLY_PREDICATE_VOCABULARY_PATH` pointing to a JSON/YAML file.
- **Effort: Small** (LLM prompt change) to **Medium** (embedding clustering pipeline).

## 3. Fact-Checking

### 3.1 Knowledge Vault (Dong et al., 2014)

**Paper:** "Knowledge Vault: A Web-Scale Approach to Probabilistic Knowledge Fusion," KDD 2014. Google.

#### Core Idea

Knowledge Vault (KV) combines extractions from web content with prior knowledge from existing KBs (Freebase) using supervised machine learning for **knowledge fusion**. It uses 15 extractors to process 1B+ web pages, producing 3B+ (subject, predicate, object) triples. Since extractors are noisy (~20% of raw extractions are correct), KV applies **data fusion** techniques:

1. **Extractor confidence:** Each extractor has a learned accuracy per predicate type.
2. **Source reliability:** Web sources are scored for trustworthiness based on the accuracy of their other claims.
3. **Prior knowledge:** Existing KB facts serve as training data and regularization.
4. **Fusion model:** A logistic regression classifier combines signals to produce a calibrated probability for each triple.

#### Algorithm

```
P(triple is correct) = σ(w_prior · prior_score +
                         Σ_e w_e · extractor_confidence_e +
                         Σ_s w_s · source_reliability_s)
```

Where `σ` is the sigmoid function and weights are learned from labeled data.

#### Relevance to Noodly

Noodly's `truth_score` (`src/noodly/models/claims.py:82-96`) is a hand-crafted multiplicative formula:

```python
truth_score = confidence * authority * recency * corroboration * conflict_penalty
```

KV's approach suggests replacing this with a **learned fusion model**:
- `confidence` → extractor confidence (already present).
- `authority` → source reliability (already present but static).
- `corroboration` → multi-source agreement (partially implemented via `len(self.evidence)`).
- **Missing:** Iterative source reliability estimation — KV updates source reliability based on the accuracy of the source's other claims.

#### Strengths
- Principled probabilistic framework with calibrated probabilities.
- Combines multiple signals (extraction, source, prior knowledge).
- Scales to web-scale data.

#### Weaknesses
- Requires labeled training data (correct/incorrect triples) for supervised learning.
- Source reliability estimation is complex at small scale.
- Google-specific infrastructure (MapReduce, proprietary extractors).

#### Recommendations for Noodly
- **Adopt the multi-signal fusion concept** but use a simpler combining function (weighted geometric mean or configurable formula) rather than a learned classifier — Noodly likely won't have enough labeled data for supervised learning.
- **Make the truth_score formula configurable** via `NOODLY_TRUTH_SCORE_FORMULA` — allow operators to choose between the current multiplicative formula, a weighted sum, or a custom expression.
- **Effort: Small** (configurable formula) to **Large** (learned fusion model).

### 3.2 TruthFinder (Yin et al., 2008)

**Paper:** "Truth Discovery with Multiple Conflicting Information Providers on the Web," IEEE TKDE, 2008.

#### Core Idea

TruthFinder solves the **truth discovery** problem: given multiple sources making conflicting claims about the same subject, determine which claims are true. It exploits a mutual reinforcement:

- A **source is trustworthy** if it provides many true claims.
- A **claim is likely true** if it is made by many trustworthy sources.

#### Algorithm

Iterative convergence:

1. **Initialize** all source trustworthiness scores uniformly.
2. **Claim scoring:** For each claim `f`, compute confidence `s(f)` as a function of the trustworthiness of sources that assert `f`:
   ```
   s(f) = Σ_{w ∈ sources(f)} t(w)
   ```
3. **Source scoring:** For each source `w`, update trustworthiness `t(w)` based on the confidence of claims it asserts:
   ```
   t(w) = Σ_{f ∈ claims(w)} s(f) / |claims(w)|
   ```
4. **Normalize** scores and repeat until convergence.
5. Introduce **implication** between claims: similar claims partially support each other (e.g., "born in 1564" and "born in April 1564").

#### Relevance to Noodly

Noodly currently assigns `source_authority` as a **static** value on `ClaimEvidence` (`src/noodly/models/claims.py:42`). TruthFinder's iterative approach would make authority **dynamic**: a source's authority increases as more of its claims are corroborated, and decreases as they are contradicted.

This maps directly to Noodly's architecture:
- Sources = unique `(author, source_type)` combinations.
- Claims = `Claim` objects in the ledger.
- The iterative update could run as part of `FactLedger.apply_decay()` or as a separate maintenance step.

#### Strengths
- Simple, intuitive algorithm.
- Handles the chicken-and-egg problem of source vs. claim reliability.
- Open-source implementations available (Python).

#### Weaknesses
- Assumes independent sources — if sources copy from each other, the algorithm over-counts evidence.
- Does not model *how wrong* a claim is — all wrong claims are equally wrong.
- Convergence is not guaranteed for all configurations.

#### Recommendations for Noodly
- **Implement a configurable TruthFinder-style iterative scoring** as an alternative to the static `truth_score`. Run the iterative update periodically (e.g., alongside `apply_decay()`).
- Make it configurable via `NOODLY_TRUTH_ALGORITHM` (`static`, `truthfinder`, `investment`).
- **Effort: Medium** — requires implementing the iterative loop and storing per-source trustworthiness scores.

### 3.3 Investment / PooledInvestment / Sums (Pasternack & Roth, 2010)

**Paper:** "Knowing What to Believe (when you already know something)," COLING 2010.

#### Core Idea

Pasternack and Roth introduced three alternative trust propagation models, all improving on TruthFinder:

**Sums:** Simple additive voting — each source contributes equally to claim confidence, and each source's trustworthiness is the sum of its claims' confidences.

**Investment:** Sources "invest" their reliability equally among their claims. Each claim's score is:
```
vote(v) = G(Σ_{s ∈ sources(v)} t(s) / |claims(s)|)
```
Where `G` is a super-linear growth function (e.g., `G(x) = x^1.2`). Sources then collect credit back proportional to their investment relative to other investors. This rewards sources that make *unique, correct* claims over sources that only repeat common knowledge.

**PooledInvestment:** Like Investment but sources pool their investments and share returns proportionally. This reduces sensitivity to the number of competing sources.

#### Key Innovation

The **prior knowledge integration framework** translates first-order logic constraints (e.g., "each person has exactly one birthdate") into a linear program that adjusts fact-finder outputs. This is applicable to Noodly: domain constraints like "each entity has at most one value for single-valued predicates" can be enforced post-scoring.

#### Relevance to Noodly

The Investment model is particularly relevant because it **rewards information diversity**: a source that provides a unique true claim not found elsewhere gets higher credit than a source that merely repeats common facts. This aligns with Noodly's need to value tacit knowledge (rare, high-value insights) more than commonly available stable facts.

#### Strengths
- Investment rewards unique knowledge, not just popular claims.
- Prior knowledge integration is principled and general.
- Well-studied with empirical results on multiple datasets.

#### Weaknesses
- More complex to implement than simple voting or TruthFinder.
- The super-linear growth function requires tuning.
- Prior knowledge must be expressed as logical constraints.

#### Recommendations for Noodly
- **Implement Investment as a `NOODLY_TRUTH_ALGORITHM` option.** The diversity-rewarding property is a natural fit for Noodly's `KnowledgeClass.tacit` claims.
- **Implement the prior knowledge framework** to enforce single-valued predicate constraints from a configurable rules file.
- **Effort: Medium** — the algorithm itself is straightforward; the prior-knowledge LP integration is harder.

### 3.4 FactScore (Min et al., 2023)

**Paper:** "FActScore: Fine-grained Atomic Evaluation of Factual Precision in Long Form Text Generation," EMNLP 2023.

#### Core Idea

FactScore evaluates factuality of LLM-generated text by:

1. **Atomic decomposition:** Break a text generation into a series of atomic facts — the smallest, self-contained units of information.
2. **Verification:** For each atomic fact, check whether it is supported by a reliable knowledge source (e.g., Wikipedia).
3. **Scoring:** FactScore = percentage of atomic facts that are supported.

The automated pipeline uses retrieval (find relevant Wikipedia passages) + a strong LLM (GPT-4) to verify each atomic fact, achieving <2% error rate compared to human evaluation.

#### Relevance to Noodly

Noodly's extraction pipeline already performs something analogous to FactScore's atomic decomposition: the `ClaimExtractor` breaks documents into structured claims (subject, predicate, object). FactScore's contribution is the *verification* step — using retrieval + LLM to check each claim against a knowledge source.

**Direct application:** After extracting claims, verify each claim against:
- The existing knowledge graph (do other claims corroborate or contradict?).
- External knowledge sources (if available and configured).

This would replace the current static `confidence` score with a verified confidence score.

#### Strengths
- Proven methodology with low error rate.
- Atomic decomposition is already done by Noodly's extractor.
- Available as a pip package (`factscore`).

#### Weaknesses
- Requires a knowledge source for verification — Noodly's KB is itself under construction.
- LLM verification calls add cost and latency.
- Designed for evaluation, not online scoring — would need adaptation for real-time use.

#### Recommendations for Noodly
- **Adopt the atomic verification concept for cross-referencing claims.** When a new claim is extracted, query the existing ledger for related claims and use an LLM to assess whether they corroborate, contradict, or are independent. This is a more principled version of the current `corroboration` factor.
- **Make verification optional and configurable** via `NOODLY_VERIFY_CLAIMS=true/false` and `NOODLY_VERIFY_MODEL` to control cost.
- **Effort: Medium** — requires an LLM verification call and integration with the existing ledger query.

### 3.5 ClaimBuster (Hassan et al., 2017)

**Paper:** "Toward Automated Fact-Checking: Detecting Check-worthy Factual Claims by ClaimBuster," KDD 2017.

#### Core Idea

ClaimBuster is an end-to-end fact-checking platform with three core components:

1. **Claim Spotter (ClaimBuster):** Uses NLP + supervised learning to identify check-worthy factual claims in text. Trained on human-labeled sentences from U.S. presidential debate transcripts. Classifies sentences as:
   - Non-factual (opinions, questions)
   - Unimportant factual claims
   - Check-worthy factual claims

2. **Claim Matcher:** Finds previously fact-checked claims similar to a new claim using semantic similarity.

3. **Knowledge Store:** Database of fact-checked claims and their verdicts.

#### Algorithm

The claim spotter uses features including:
- Sentiment scores
- Named entity presence and types
- POS tag patterns
- TF-IDF features
- Sentence position and length

A classifier (SVM, later neural models) predicts check-worthiness.

#### Relevance to Noodly

Noodly extracts *all* factual claims from documents. ClaimBuster's check-worthiness scoring could be used to **prioritize** which claims deserve human review or deeper verification:

- Claims with high check-worthiness → queue for `owner_confirmed` review.
- Claims with low check-worthiness → auto-promote if score is sufficient.

The **Claim Matcher** component is also relevant: when a new claim arrives, search for previously verified claims to inherit their status.

#### Strengths
- Real-world deployment (monitored U.S. debates and Hansard live).
- The check-worthiness concept is directly applicable to Noodly's promotion workflow.
- Published API available.

#### Weaknesses
- Trained on political discourse; may not generalize to enterprise knowledge.
- The claim spotter only classifies sentences, not structured triples.
- The matching component is basic (TF-IDF similarity).

#### Recommendations for Noodly
- **Adopt the check-worthiness concept** as a `check_worthiness: float` field on `Claim`. Use a simple heuristic initially: claims about entities, relationships, and processes are more check-worthy than metadata claims.
- **Use check-worthiness to prioritize human review queues** and configure auto-promotion thresholds per worthiness tier via `NOODLY_AUTO_PROMOTE_CHECK_WORTHY_THRESHOLD`.
- **Effort: Small** — the heuristic is simple; a trained model can come later.

### 3.6 FEVER Dataset and Related Benchmarks

**Paper:** "FEVER: a Large-scale Dataset for Fact Extraction and VERification," NAACL 2018.

#### Overview

FEVER (Fact Extraction and VERification) is the standard benchmark for automated fact verification:

- **185,445 claims** generated by altering Wikipedia sentences.
- **Labels:** SUPPORTED, REFUTED, NOT ENOUGH INFO.
- **Evidence:** For SUPPORTED/REFUTED claims, annotators recorded the Wikipedia sentence(s) forming evidence.
- **Knowledge source:** A 2017 Wikipedia dump (~5M articles).

**Task pipeline:**
1. **Document retrieval:** Find relevant Wikipedia articles for a claim.
2. **Sentence selection:** Select evidence sentences from those articles.
3. **Claim classification:** Classify as SUPPORTED, REFUTED, or NOT ENOUGH INFO.

**Best systems:** Achieve ~80% accuracy on the full pipeline (retrieval + classification). The BEVERS system (2023) holds the highest FEVER score among published systems.

#### Related Benchmarks
- **SciFact:** Fact verification for scientific claims against paper abstracts.
- **LIAR:** Political statement fact-checking with 6 fine-grained labels.
- **MultiFC:** Multi-domain fact-checking from 26 fact-checking websites.
- **AVerITeC:** Verification of image-text claims with web evidence (FEVER 2025 shared task).

#### Relevance to Noodly

FEVER's pipeline directly maps to Noodly's needs:
- Document retrieval → Graphiti's hybrid search (`Brain.search_facts()` in `src/noodly/graph/brain.py:118`).
- Evidence selection → Finding supporting/refuting `ClaimEvidence`.
- Claim classification → Noodly's `ClaimStatus` progression.

**Key insight:** FEVER shows that combining retrieval + classification achieves much better results than classification alone. Noodly should always retrieve related claims before scoring a new one.

#### Recommendations for Noodly
- **Use FEVER-style retrieval-then-classify for claim verification.** Before scoring a new claim, retrieve top-k related claims from the ledger (via embedding search), then use an LLM to classify the relationship (supports, refutes, independent).
- **Consider evaluating Noodly's claim extraction quality** using a FEVER-inspired benchmark: generate test claims from known documents and measure extraction precision/recall.
- **Effort: Medium** (retrieval-then-classify) to **Large** (custom benchmark).

### 3.7 DeepDive / Knowledge Base Construction

**System:** [http://deepdive.stanford.edu/](http://deepdive.stanford.edu/) (Stanford, CACM 2017)

> **Note:** DeepDive is in maintenance mode since 2017. Its successor is the [Snorkel](https://snorkel.ai/) project.

#### Core Idea

DeepDive frames knowledge base construction (KBC) as a single large statistical inference problem using a **factor graph**. The system:

1. **Extraction:** UDFs (user-defined functions) extract candidate entities and relationships from text.
2. **Factor graph construction:** Users write SQL-like rules that define factors connecting extracted variables. Each factor represents a domain constraint or feature.
3. **Weight learning:** DeepDive learns factor weights from labeled data or distant supervision.
4. **Probabilistic inference:** Gibbs sampling computes marginal probabilities for each extracted fact.

**Key innovation:** Treating ETL as inference. Instead of building a pipeline of independent extraction, cleaning, and integration steps (each with its own error budget), DeepDive jointly optimizes all steps as a single probabilistic model.

#### Relevance to Noodly

DeepDive's approach is philosophically aligned with Noodly's goal of building a "Company Brain" from unstructured data. Key parallels:

| DeepDive Concept | Noodly Equivalent |
|-----------------|-------------------|
| Candidate extraction | `ClaimExtractor.extract()` |
| Factor graph variables | `Claim` objects |
| Factor functions | Domain constraints (e.g., "a person has one employer at a time") |
| Marginal probability | `truth_score` |
| Distant supervision | Using existing KB facts as training labels |

**What Noodly could adopt:**
- **Distant supervision:** Use high-confidence existing claims (status `canonical`) as positive training examples for the extractor.
- **Domain constraints as factors:** Express constraints like "single-valued predicates" as scoring adjustments (similar to Pasternack & Roth's prior knowledge framework).
- **Joint inference:** Instead of scoring claims independently, consider inter-claim dependencies (if A is true and A contradicts B, then B is less likely).

#### Strengths
- Joint optimization improves quality compared to pipeline approaches.
- Factor graphs are expressive and well-understood.
- Proven in real-world applications (paleobiology, genomics, anti-human trafficking).

#### Weaknesses
- Maintenance mode — no new development.
- Complex infrastructure (Postgres + custom sampler).
- Factor graph construction requires domain expertise.
- Gibbs sampling is slow for large graphs.

#### Recommendations for Noodly
- **Do not adopt DeepDive directly** — it is unmaintained and heavyweight.
- **Adopt the distant supervision pattern:** Periodically fine-tune the extraction prompt using `canonical` claims as examples of good extractions.
- **Adopt the constraint-as-factor concept:** Implement configurable domain constraints that adjust `truth_score` (e.g., single-valued predicates, temporal consistency).
- **Effort: Small** (distant supervision in prompt) to **Large** (full factor graph inference).

## 4. Conflicting Fact Management

### 4.1 Dung's Abstract Argumentation Frameworks (1995)

**Paper:** "On the acceptability of arguments and its fundamental role in nonmonotonic reasoning, logic programming and n-person games," Artificial Intelligence, 1995.

#### Core Concept

An **argumentation framework** is a pair `AF = (Args, attacks)` where:
- `Args` is a set of arguments.
- `attacks ⊆ Args × Args` is a binary attack relation.

An argument `A` attacks argument `B` if `(A, B) ∈ attacks`. The central question: which sets of arguments are *acceptable* (can be collectively believed)?

#### Semantics

**Definitions:**
- A set `S ⊆ Args` is **conflict-free** if no argument in `S` attacks another in `S`.
- An argument `A` is **defended by** `S` if every attacker of `A` is attacked by some member of `S`.
- A set `S` is **admissible** if it is conflict-free and defends all its members.

**Four classical semantics:**

| Semantics | Definition | Intuition |
|-----------|-----------|-----------|
| **Grounded** | Smallest (unique) complete extension | Conservative: only accept arguments that *must* be accepted |
| **Preferred** | Maximal (w.r.t. set inclusion) admissible sets | Credulous: accept as many arguments as possible without conflict |
| **Stable** | Conflict-free sets that attack every non-member | Opinionated: every argument is either accepted or attacked |
| **Complete** | Admissible sets that include all arguments they defend | Between grounded and preferred |

#### Mapping to Noodly

Noodly's claims can be modeled as arguments. Two claims attack each other if they are contradictory (e.g., "CEO of Acme is Alice" and "CEO of Acme is Bob" for a single-valued predicate):

| AF Concept | Noodly Mapping |
|-----------|----------------|
| Argument | `Claim` |
| Attack | Contradictory claims (same subject+predicate, different object, for single-valued predicates) |
| Grounded extension | The minimal set of claims that are non-controversial |
| Preferred extension | The maximal consistent set of claims — the "best" working truth |
| Stable extension | All claims are either accepted or have a reason for rejection |

The `conflict_penalty` in `truth_score` (`src/noodly/models/claims.py:86-89`) is a simplistic version of this: each piece of refuting evidence reduces the score by 30%. Dung's framework offers a more principled approach.

#### Strengths
- Provides formal semantics for "what to believe when things conflict."
- Multiple semantics for different use cases (conservative vs. credulous).
- Well-studied with extensive tooling (e.g., [ASPARTIX](http://www.dbai.tuwien.ac.at/research/project/argumentation/systempage/), [ConArg](http://www.inteligentes.xyz/ConArg/)).

#### Weaknesses
- Abstract — does not define *what* arguments are or *why* they attack each other. Noodly must define the attack relation.
- Computational complexity: computing preferred/stable extensions is NP-hard in general. (Manageable for Noodly's scale.)
- Does not handle weighted arguments or varying confidence.

#### Recommendations for Noodly
- **Implement a conflict detection layer** that identifies attack relations between claims. Two claims attack each other if:
  1. They share a subject and predicate but have different objects.
  2. They have contradictory predicates (e.g., "is active" vs. "is decommissioned").
  3. They violate a configurable domain constraint (e.g., single-valued predicate).
- **Use grounded semantics for conservative defaults** (only accept uncontested claims) and **preferred semantics for the full working truth** (accept the maximum consistent set, preferring higher-scored claims).
- **Make the semantics configurable** via `NOODLY_CONFLICT_SEMANTICS` (`grounded`, `preferred`, `stable`).
- **Effort: Medium** — requires implementing the labeling algorithm and defining the attack relation.

### 4.2 AGM Belief Revision (Alchourrón, Gärdenfors, Makinson, 1985)

**Paper:** "On the Logic of Theory Change: Partial Meet Contraction and Revision Functions," Journal of Symbolic Logic, 1985.

#### Core Concept

AGM theory defines three operations on a **belief set** `K` (a logically closed set of propositions):

1. **Expansion** `K + φ`: Add a new belief `φ` to `K` and close under logical consequence. No beliefs are removed. This is straightforward when `φ` is consistent with `K`.

2. **Contraction** `K - φ`: Remove belief `φ` from `K` while retaining as many other beliefs as possible. The result is a **maximal subset** of `K` that does not imply `φ`.

3. **Revision** `K * φ`: Add `φ` to `K`, ensuring consistency. This may require removing some existing beliefs. The Levi identity connects revision to contraction: `K * φ = (K - ¬φ) + φ`.

#### Postulates (for revision `K * φ`)

1. **Closure:** `K * φ` is logically closed.
2. **Success:** `φ ∈ K * φ`.
3. **Inclusion:** `K * φ ⊆ K + φ`.
4. **Vacuity:** If `¬φ ∉ K`, then `K * φ = K + φ`.
5. **Consistency:** `K * φ` is consistent (if `φ` is consistent).
6. **Extensionality:** If `φ ↔ ψ`, then `K * φ = K * ψ`.

The **Harper identity** connects contraction to revision: `K - φ = K ∩ (K * ¬φ)`.

#### Relevance to Noodly

Noodly's claim lifecycle maps to AGM operations:
- **Adding a new claim** (non-contradictory) → **Expansion**: the ledger grows.
- **Adding a contradictory claim** → **Revision**: the new claim should be accepted (if sufficiently authoritative), and the old claim should be superseded. Noodly currently does this via `ClaimStatus.superseded`, but without the principled "minimal change" guarantee.
- **Retracting a claim** → **Contraction**: removing a claim while preserving as many dependent beliefs as possible.

**Key insight for Noodly:** AGM's principle of **minimal change** — when revising beliefs, change as little as possible. Currently, Noodly doesn't track dependencies between claims, so it can't know what else might need to change when a claim is superseded.

#### Strengths
- Gold-standard formal framework for belief change.
- Minimal change principle is intuitively correct and well-axiomatized.
- Connects contraction and revision through clean identities.

#### Weaknesses
- Assumes beliefs are sentences in a formal logic — Noodly's claims are structured triples, not logical formulas.
- Does not handle iterated revision well (the original framework is for a single revision step). Darwiche & Pearl (1997) address this.
- Assumes a single agent; Noodly has multiple sources.

#### Recommendations for Noodly
- **Adopt the minimal change principle** as a design guideline: when a claim is superseded, only update directly affected claims, not the entire ledger. Track claim dependencies (which claims were derived from or influenced by the superseded claim).
- **Implement a `Claim.depends_on: list[UUID]` field** to track derivation chains, enabling dependency-directed revision.
- **Do not implement full AGM logic** — the formalism is too heavyweight. Instead, use the postulates as design constraints for the supersession logic.
- **Effort: Small** — design principle adoption + `depends_on` field.

### 4.3 Truth Maintenance Systems (TMS)

#### JTMS (Justification-based TMS) — Doyle, 1979

A JTMS maintains a set of **nodes** (beliefs) and **justifications** (dependencies). Each node is labeled **IN** (believed) or **OUT** (not believed). A justification specifies:
- A set of **in-list** nodes (must be IN for the justification to hold).
- A set of **out-list** nodes (must be OUT for the justification to hold).

When a node's status changes, the JTMS propagates the change through the dependency network using **dependency-directed backtracking** — when a contradiction is found, the system traces back through justifications to find the responsible assumption and retracts it.

#### ATMS (Assumption-based TMS) — de Kleer, 1986

The ATMS improves on JTMS by:
- Tracking **all possible consistent sets of assumptions** (environments) simultaneously.
- Associating each node with the set of environments in which it is believed.
- No retraction is ever needed — the system simply manages which environments are consistent.
- **Context switching is free** — to see what's believed under a different set of assumptions, just select a different environment.

#### Relevance to Noodly

TMS concepts map directly to Noodly's claim management:

| TMS Concept | Noodly Mapping |
|-------------|----------------|
| Node | `Claim` |
| Justification | `ClaimEvidence` (the evidence that supports the claim) |
| IN/OUT label | `ClaimStatus` (active statuses vs. `superseded`/`rejected`) |
| Assumption | Claims from a specific source (axioms that could be wrong) |
| Environment (ATMS) | A consistent subset of claims (one possible "world view") |

**Key insight:** The ATMS's ability to maintain multiple consistent environments simultaneously is powerful for Noodly: instead of committing to one "truth," the system could maintain multiple candidate world views and let the consumer choose.

#### Strengths
- JTMS provides dependency-directed backtracking — when a claim is retracted, only affected downstream claims need updating.
- ATMS avoids retraction entirely — all views are maintained, and switching is free.
- Well-understood algorithms with polynomial-time complexity for JTMS.

#### Weaknesses
- ATMS can have exponentially many environments (2^n assumptions).
- Both systems assume binary belief (IN/OUT), not graded confidence.
- Implementation complexity is non-trivial.

#### Recommendations for Noodly
- **Adopt JTMS-style dependency tracking.** When evidence for a claim is retracted, propagate the change: any claims that were promoted based on that evidence should be re-evaluated.
- **This requires the `depends_on` field** recommended in the AGM section, extended with justification metadata (which evidence supports each dependency link).
- **Do not implement a full ATMS** — the exponential environments are impractical. But the *concept* of multiple world views could be useful for future "what-if" analysis features.
- **Effort: Medium** — dependency tracking is moderately complex; a full JTMS propagation engine is significant work.

### 4.4 Defeasible Reasoning / Non-Monotonic Logic

#### Core Concept

In classical (monotonic) logic, adding new information never invalidates existing conclusions. In **non-monotonic logic**, conclusions can be **defeated** by new information:

> "Birds fly. Tweety is a bird. Therefore, Tweety flies."
> "Penguins don't fly. Tweety is a penguin." → Retract: Tweety flies.

**Key formalisms:**
- **Default Logic (Reiter, 1980):** Rules have prerequisites, justifications, and conclusions. A default rule applies if its prerequisite is true and its justification is consistent with current beliefs.
- **Circumscription (McCarthy, 1980):** Minimize the extension of abnormality predicates — assume things are normal unless proven otherwise.
- **Answer Set Programming (ASP):** Logic programs with negation-as-failure semantics. Stable models (answer sets) represent consistent world views.
- **Defeasible Logic (Nute, 1991):** Explicitly represents strict rules, defeasible rules, and defeater rules. A rule hierarchy determines which rules override others.

#### Relevance to Noodly

Noodly's knowledge classes already embody a form of defeasible reasoning:

| Knowledge Class | Defeasibility |
|----------------|---------------|
| `stable` | Low — these facts should rarely be overridden |
| `process` | Medium — processes change over time |
| `tacit` | High — informal knowledge is easily outdated |
| `stateful` | Very high — current state is inherently volatile |

The `KnowledgeClass` determines the `DECAY_RATES` in `src/noodly/scoring/ledger.py:19-24`, which is a simple form of defeasibility: stateful claims lose relevance fastest.

**What's missing:** There's no mechanism for a new claim to *explicitly defeat* an older claim. Currently, a new contradictory claim just accumulates alongside the old one, and the `conflict_penalty` reduces both scores. In a defeasible reasoning system, the newer, more specific, or higher-authority claim would *defeat* the older one, marking it as overridden.

#### Strengths
- Natural fit for knowledge that changes over time.
- Rule hierarchies allow explicit modeling of override relationships.
- ASP solvers are efficient for moderate-scale reasoning.

#### Weaknesses
- Complex formalism; steep learning curve.
- Requires explicit rule definition — who writes the override rules?
- Most implementations are for logic programs, not knowledge graphs.

#### Recommendations for Noodly
- **Implement explicit defeater relationships** between claims. When a new claim contradicts an existing one and has higher authority or more recent evidence, mark the old claim as `superseded` with a `defeated_by: UUID` reference.
- **Use the `KnowledgeClass` hierarchy** to determine default defeater priority: more volatile classes are more easily defeated.
- **Make the defeat rules configurable** via a `NOODLY_DEFEAT_RULES` config file, allowing operators to specify override priorities per predicate type.
- **Effort: Small** (defeat relationship + field) to **Medium** (configurable rules engine).

### 4.5 Master Data Management (MDM) Conflict Resolution

#### Overview

Enterprise MDM systems handle conflicting records from multiple source systems by constructing a **golden record** — the single best version of an entity. Key concepts:

#### Survivorship Rules

| Rule Type | How It Works | Example |
|-----------|-------------|---------|
| **Source ranking** | Trust sources in a configured priority order | CRM > Email > Web scraping |
| **Most recent** | Prefer the most recently updated value | Use the latest address |
| **Most frequent** | Prefer the value that appears most often | Most common spelling of a name |
| **Most complete** | Prefer the record with the most non-null fields | — |
| **Longest** | Prefer the longest string value | Full name over abbreviation |
| **Decay** | Source trust decreases over time | A 3-year-old CRM record is less trustworthy than a 1-year-old one |
| **Custom function** | Domain-specific logic | "If the record is from region X, prefer local tax ID format" |

#### Implementations

- **Informatica MDM:** Source ranking + decay-based trust scores. Decay rate is configurable per source system. When trust decays, the system re-evaluates which source should provide the golden value.
- **Reltio:** Operational Value (OV) rules. Each attribute can have a different survivorship strategy. Supports `Most Recent Source`, `Most Frequent`, `Longest`, `Max/Min`, and custom functions. Tracks all contributing values for audit.
- **Tamr:** ML-driven record consolidation. Uses "most common value" as the default, with configurable overrides per attribute.

#### Relevance to Noodly

MDM survivorship rules are **directly applicable** to Noodly's conflict resolution. The current system has implicit survivorship:
- `authority` is a static source-level score (analogous to source ranking).
- `recency` decay (`src/noodly/scoring/ledger.py:163-180`) is analogous to MDM decay.
- `truth_score` combines these into a single value (analogous to computing the golden record).

**What's missing:**
- No per-attribute/per-predicate survivorship rules — all claims use the same formula.
- No "most frequent" or "most complete" strategies.
- No audit trail of why a particular claim "won" over another.
- No configurable source ranking per topic (the user has specifically requested topic-specific authority).

#### Strengths
- Battle-tested in enterprise environments at massive scale.
- Intuitive, business-friendly rule configuration.
- Audit trails and explainability built in.

#### Weaknesses
- MDM systems assume structured records with known schemas — Noodly's claims are more dynamic.
- Rule configuration can become complex with many exceptions.
- No probabilistic semantics — rules are deterministic.

#### Recommendations for Noodly
- **Implement configurable survivorship strategies per predicate type.** For single-valued predicates (e.g., "CEO of Acme"), configure which strategy determines the winning claim: source ranking, most recent, highest confidence, or custom.
- **Add topic-specific source authority** as the user has requested: allow configuring authority per `(source_type, topic)` pair via a `NOODLY_AUTHORITY_CONFIG_PATH` pointing to a YAML file.
- **Record the "winning reason"** when a claim supersedes another — store why it won (higher authority, more recent, more evidence) for audit.
- **Effort: Medium** — requires a `SurvivorshipStrategy` abstraction and per-predicate configuration.

### 4.6 CRDTs and Eventual Consistency

#### Core Concept

**Conflict-free Replicated Data Types (CRDTs)** are data structures designed for replicated systems where:
1. Any replica can be updated independently, concurrently, and without coordination.
2. All replicas are guaranteed to **eventually converge** to the same state.
3. An algorithm automatically resolves any inconsistencies — there is *no concept of conflict*.

**Formally defined** by Shapiro, Preguiça, Baquero, and Zawirski (2011).

**Two approaches:**
- **State-based (CvRDTs):** Replicas send their full state; states are merged using a **join** operation on a join-semilattice (commutative, associative, idempotent).
- **Operation-based (CmRDTs):** Replicas send operations; operations must be commutative (or delivered in causal order).

**Common CRDTs:**
- **G-Counter:** Grow-only counter (each node has its own counter; merge = take max per node).
- **PN-Counter:** Positive-negative counter (two G-Counters).
- **G-Set:** Grow-only set (merge = union).
- **OR-Set (Observed-Remove Set):** Supports add and remove with unique tags; "add wins" on concurrent add/remove.
- **LWW-Register (Last-Writer-Wins):** Each value has a timestamp; merge = take the value with the latest timestamp.

#### Relevance to Noodly

CRDTs provide interesting patterns for knowledge systems:

| CRDT Pattern | Noodly Application |
|-------------|-------------------|
| **LWW-Register** | For single-valued predicates, the most recent claim wins — this is the "most recent" survivorship rule from MDM, with convergence guarantees. |
| **OR-Set** | For multi-valued predicates (e.g., "tags of entity X"), concurrent additions from different sources should all be preserved. |
| **G-Counter** | For counting evidence (corroboration count) — each source's contribution is tracked independently, merge = max per source. |
| **Causal ordering** | Claims should be ordered by causal relationships (derived-from), not just timestamps. |

**Key insight:** The CRDT principle of "no conflict by design" is aspirational for Noodly. By carefully choosing the merge semantics for each predicate type, many apparent conflicts can be resolved automatically:
- Single-valued predicates → LWW-Register (most recent wins)
- Multi-valued predicates → OR-Set (union)
- Numeric aggregates → PN-Counter

#### Strengths
- Mathematically proven convergence — no conflict resolution needed.
- Well-suited for distributed and eventually-consistent systems.
- Composable: complex CRDTs can be built from simpler ones.

#### Weaknesses
- LWW semantics can silently lose data (a concurrent older update is discarded).
- Not all knowledge conflicts are resolvable by data-structure choice — some require semantic understanding.
- CRDTs assume operations are well-defined; knowledge claims are more ambiguous.

#### Recommendations for Noodly
- **Adopt LWW-Register semantics for single-valued predicates** as the default conflict resolution strategy. When two claims conflict on a single-valued predicate, the most recent one wins (already partially implemented via `recency` scoring).
- **Adopt OR-Set semantics for multi-valued predicates** — all values are preserved, tagged by source.
- **Mark these as the default strategies** in the configurable survivorship framework, overridable per predicate.
- **Effort: Small** — this is primarily a design decision applied to the survivorship strategy framework.
